Create a Website with AppCrane

Go from zero to a live website with HTTPS in under 10 minutes.
This guide walks you through every step — no prior AppCrane experience needed.

How It Works

Push code to GitHub → AppCrane deploys it → Caddy adds HTTPS → Live at your domain

AppCrane runs on your server and manages the full lifecycle: cloning your repo, building, starting the process, health checks, and routing traffic via Caddy with automatic Let's Encrypt certificates.

Every app gets two environments — production and sandbox — running simultaneously on separate ports so you can test before going live.

Prerequisites

  • An Ubuntu server (22.04+ recommended) with root/sudo access
  • A domain name pointed to your server's IP (A record)
  • Your app code in a GitHub repository
  • Node.js 18+ installed on the server

Step 1 — Install AppCrane on Your Server

1 SSH into your server and clone AppCrane:

ssh you@your-server
git clone https://github.com/gitayg/appCrane.git ~/appCrane
cd ~/appCrane
npm install
npm link          # makes the 'crane' CLI available globally

2 Initialize the admin account:

crane init --name admin --email you@example.com

This creates your admin API key and saves it to ~/.appcrane/config.json.

3 Start the AppCrane server:

npx pm2 start server/index.js --name appcrane
npx pm2 save
npx pm2 startup   # auto-start on reboot

Step 2 — Prepare Your App

Add a deployhub.json manifest to the root of your repo:

{
  "name": "My Website",
  "version": "1.0.0",
  "fe": {
    "build": "npm run build",
    "serve": "npx serve -s dist -l $PORT"
  },
  "be": {
    "entry": "node server.js",
    "health": "/api/health"
  }
}
Frontend-only site? Omit the "be" section. AppCrane will build and serve just the frontend.
Backend-only API? Omit the "fe" section. Just provide "be" with your entry point.

If your app has a backend, expose a health endpoint:

// server.js (Express example)
app.get('/api/health', (req, res) => {
  res.json({ status: 'ok', version: '1.0.0' });
});

Step 3 — Create the App in AppCrane

Back on your server, register your app:

crane app create \
  --name "My Website" \
  --slug mysite \
  --domain mysite.example.com \
  --repo https://github.com/yourname/mysite

This allocates ports, creates the directory structure at /data/apps/mysite/, and configures routing.

OptionDescription
--nameDisplay name (shown in dashboard)
--slugURL-safe identifier — your site lives at /slug/
--domainCustom domain (Caddy auto-provisions HTTPS)
--repoGitHub repo URL to deploy from

Step 4 — Set Environment Variables

If your app needs secrets (API keys, DB URLs), set them securely:

crane env set mysite --env production \
  DATABASE_URL=postgres://user:pass@host/db \
  STRIPE_KEY=sk_live_xxxx

Variables are encrypted with AES-256-GCM at rest. Even the admin cannot read them — only the running app process can.

Note: Set env vars for both production and sandbox if both environments need them.

Step 5 — Deploy

Deploy to sandbox first to test:

crane deploy mysite --env sandbox

AppCrane will:

  1. Clone your repo (latest main branch)
  2. Run npm install
  3. Run the build command from deployhub.json
  4. Write encrypted env vars to .env
  5. Start the process via PM2
  6. Run the health check
  7. Route traffic to the new release (zero downtime)

Once sandbox looks good, promote to production:

crane promote mysite

Or deploy directly to production:

crane deploy mysite --env production

Step 6 — Set Up Auto-Deploy (Optional)

Configure a GitHub webhook so every push to main triggers a deploy automatically:

  1. Get your webhook URL and secret: 
    crane webhook show mysite
  2. In GitHub: Settings → Webhooks → Add webhook
    • Payload URL: https://crane.example.com/api/webhooks/<token>
    • Content type: application/json
    • Secret: the secret from step 1
    • Events: Just the push event

Now every git push to main triggers an automatic sandbox deploy.

Step 7 — Monitor & Manage

Health Checks

AppCrane pings your health endpoint every 30 seconds. If it fails, you get an email alert and the app auto-restarts after consecutive failures.

crane health test mysite --env production
crane health config mysite --interval 60 --threshold 3

Rollback

Something broke? Roll back to the previous release instantly:

crane rollback mysite --env production

AppCrane keeps the last 5 releases for instant rollback.

View Logs

crane logs mysite --env production --lines 100

Dashboard

Open the web dashboard for a visual overview of all apps, health status, deploy history, and more:

https://crane.example.com/dashboard

Using the REST API

Every operation above is also available via HTTP. Pass your API key in the X-API-Key header:

# Deploy via curl
curl -X POST https://crane.example.com/api/apps/mysite/deploy/sandbox \
  -H "X-API-Key: dhk_user_xxxxx"

# Check health
curl https://crane.example.com/api/apps/mysite/health \
  -H "X-API-Key: dhk_user_xxxxx"

# List all apps
curl https://crane.example.com/api/apps \
  -H "X-API-Key: dhk_user_xxxxx"

Adding Team Members

Create a user and assign them to specific apps:

crane user create --name alice --email alice@example.com
crane app assign mysite --email alice@example.com

Each user gets their own API key scoped to their assigned apps. They can deploy, set env vars, and manage health — but cannot create or delete apps.

ActionAdminApp User
Create / delete appsYesNo
Deploy / rollback / promoteNoYes (own apps)
View / edit env varsNoYes (own apps)
Manage usersYesNo
View audit logYesYes (own apps)

App Directory Structure

/data/apps/mysite/
  production/
    current -> releases/20260412-143022   # symlink to active release
    releases/
      20260412-143022/                    # each deploy is a timestamped folder
      20260411-091500/
    shared/                               # persistent data across deploys
  sandbox/
    current -> releases/...
    releases/
    shared/

Troubleshooting

ProblemSolution
Deploy fails at build stepCheck that deployhub.json build command works locally with npm run build
Health check fails after deployEnsure your app listens on $PORT (AppCrane sets this env var)
Assets load with wrong pathSet base in your Vite/webpack config to process.env.APP_BASE_PATH
HTTPS not workingVerify your domain's A record points to the server IP, then restart Caddy
Permission deniedApp users can only access their assigned apps. Ask admin to run crane app assign
